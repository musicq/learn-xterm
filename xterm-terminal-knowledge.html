01

Mental Model

First separate display, command interpretation, and process management.

xterm.js

What it is

xterm.js is a terminal emulator that runs in the browser. Think of it as a terminal screen plus keyboard input layer: it renders characters, colors, cursors, scrollback, and exposes user input as a data stream.

not shell

What it is not

It is not a shell. It does not execute ls, cd, or npm run dev, and it does not access the file system or manage real processes.

boundary

The one sentence to remember

xterm.js handles terminal-like display and input; the shell understands commands and produces results; the PTY makes the shell believe it is attached to a real terminal.

Layer Responsible for Not responsible for
xterm.js Rendering characters, colors, cursors; parsing ANSI; listening for input; maintaining scrollback. Executing commands, starting processes, understanding cd, reading the file system.
Frontend app Creating Terminal and WebSocket instances, bridging I/O, syncing resize events. After connecting a real PTY, it should not echo input or parse complex control keys.
node-pty Creating a pseudo-terminal, writing input to the shell, reading shell output back. It does not render text or understand command semantics.
shell Parsing commands, maintaining cwd/env, generating prompts, history/completion, starting programs. It does not draw the terminal UI in the browser.
02

Input / Output

A terminal is a cursor model, not a normal HTML text flow.

Minimal initializationbrowser
import { Terminal } from "@xterm/xterm";
import "@xterm/xterm/css/xterm.css";

const dom = document.getElementById("terminal");
if (!dom) throw new Error("#terminal not found");

const term = new Terminal();
term.open(dom);
term.write("Hello xterm.js\r\n$ ");
write and line breakscursor model
term.write("A");
term.write("B\n");
term.write("C");
term.write("D\r\n");
term.write("E");

// Result is roughly:
// AB
//   CD
// E
Input Common onData value How to think about it while learning
Printable characters "a"、"1" In a fake echo terminal you may call term.write(data); after connecting a PTY, send it to the backend unchanged.
Enter "\r" In a frontend fake shell, turn it into \r\n$ ; with a real PTY, write it directly to the PTY.
Backspace "\x7f" A fake shell can visually erase with "\b \b", while also updating the command buffer.
Ctrl+C "\x03" A fake shell may show ^C; a real PTY passes it to the shell/process.
Arrow keys "\x1b[A", etc. Real shell/readline/zle handles history and line editing. The frontend should not parse it itself.
Core pitfall: screen content is not program state.

In a fake terminal, seeing abc on screen does not mean JavaScript knows the current command is abc. You must maintain command, clear it after Enter, and update both state and screen on Backspace.

03

Shell and PTY

A PTY is not mainly about multiple connections. It gives the shell real terminal semantics.

A shell is a normal program

zsh, bash, fish, and powershell are shells. They parse commands, generate prompts, maintain the current directory, and start child processes.

A PTY is a pseudo-terminal device

A PTY is like a pipe disguised as a real terminal. The shell believes it is attached to a terminal, so it enables interactive mode, colors, history, fullscreen programs, and job control.

child_process is not enough

child_process is good for one-shot commands. Long-lived interactive sessions, vim/top/less, Ctrl+C, resize, and arrow-key history strongly call for a PTY.

Scenario Better fit Why
Run git status and only need final text child_process One-shot task; no terminal semantics needed.
Open a long-lived zsh session in the browser PTY Needs prompt, history, completion, and interactive input.
Run vim, top, or less PTY Fullscreen programs depend on rows/cols, cursor control, and raw mode.
Stream npm install logs Depends If you only need logs, child_process can work; for terminal-like progress and colors, PTY is closer.
Minimal node-pty structureserver.js
import pty from "node-pty";

const shell = process.env.SHELL || "/bin/zsh";

const ptyProcess = pty.spawn(shell, [], {
  name: "xterm-color",
  cols: 80,
  rows: 24,
  cwd: process.cwd(),
  env: process.env,
});

ptyProcess.onData((data) => {
  socket.send(data);
});

socket.addEventListener("message", (event) => {
  ptyProcess.write(event.data.toString());
});

socket.addEventListener("close", () => {
  ptyProcess.kill();
});
After connecting a real PTY, the frontend should not echo input.

When the user presses a, xterm.js fires onData("a"), and the frontend sends it to the PTY. After TTY/shell echo, the output returns through WebSocket to term.write. If the frontend also calls term.write(data), you may see aa.

04

Minimal Web Terminal

The frontend only bridges data; the backend owns the PTY session.

Frontend bridgemain.js
const ws = new WebSocket("ws://localhost:3001");
const fitAddon = new FitAddon();
const term = new Terminal({ scrollback: 1000 });

term.loadAddon(fitAddon);
term.open(container);
fitAddon.fit();

term.onData((data) => {
  if (ws.readyState === WebSocket.OPEN) {
    ws.send(JSON.stringify({ type: "input", data }));
  }
});

ws.addEventListener("message", (event) => {
  term.write(event.data);
});

function sendResize() {
  if (ws.readyState === WebSocket.OPEN) {
    ws.send(JSON.stringify({
      type: "resize",
      cols: term.cols,
      rows: term.rows,
    }));
  }
}
Backend protocolserver.js
socket.addEventListener("message", (event) => {
  try {
    const payload = JSON.parse(event.data.toString());

    if (payload.type === "input") {
      ptyProcess.write(payload.data);
    }

    if (payload.type === "resize") {
      ptyProcess.resize(payload.cols, payload.rows);
    }
  } catch (error) {
    console.error("invalid ws message", error);
  }
});

FitAddon

fitAddon.fit() computes how many columns and rows the frontend terminal should have. It must run after term.open(container).

Resize sync

Frontend fitting only changes xterm.js dimensions. To make vim and top render correctly, send term.cols and term.rows to backend pty.resize.

Resource cleanup

The frontend removes resize listeners and calls term.dispose(), fitAddon.dispose(), and ws.close(). The backend calls ptyProcess.kill() after socket close.

05

Deeper Topics

ANSI, paste behavior, raw/cooked mode, and echo are all part of how terminals really work.

Sequence Meaning How to remember it
\x1b[31m Set foreground color to red ESC + SGR parameters. It remains active until reset.
\x1b[0m Reset styling Colored CLI output is often “set color, write text, reset.”
\x1b[2J Clear screen Often paired with \x1b[H.
\x1b[H Move cursor to top-left Fullscreen programs work by moving the cursor and repainting regions.
\x1b[K Clear current line from cursor to end Common during prompt redraws and history navigation.
\x1b[?2004h Enable bracketed paste The shell asks the terminal to wrap pasted content.
\x1b[200~...\x1b[201~ Pasted content boundary The frontend should not split it; pass it unchanged to the PTY.

ANSI is stateful

After term.write("\x1b[31mred "), later text stays red unless \x1b[0m resets it. A terminal is more like “current brush state” than HTML element styling.

Clearing is not moving home

\x1b[2J clears the screen; \x1b[H moves the cursor. To write at the top-left, use \x1b[2J\x1b[H.

copy / paste

Copy usually relies on browser selection and clipboard behavior. Paste enters the xterm.js input stream, then goes through onData to the PTY. Multi-line paste may arrive as one chunk wrapped in bracketed paste markers.

cooked mode

TTY/shell buffers a line and handles echo, Backspace, and Ctrl+C. Common with normal shell prompts and simple commands.

raw mode

The program receives key data as directly as possible and handles arrows, Ctrl+C, and screen repainting itself. Common with vim, top, and htop.

06

Debug Checklist

When pressing a does not display anything, do not guess. Walk the chain layer by layer.

Did frontend term.onData receive "a"?
Did frontend ws.send send { type: "input", data: "a" }?
Did backend WebSocket message receive the payload?
Did the backend call ptyProcess.write("a")?
Did backend ptyProcess.onData receive echoed output?
Did backend socket.send send the output back to the browser?
Did frontend WebSocket message receive output?
Did the frontend call term.write(output)?
Are xterm.js DOM size, focus, and dispose state normal?
Symptom First suspect Explanation
Typed characters appear twice Double echo from frontend and shell After PTY connection, the frontend should not call term.write(data).
Enter does not move to the start of the next line Only \n was used, or only \r was returned Terminals commonly use \r\n for “start of next line.”
vim/top display is misaligned Frontend and backend rows/cols do not match After FitAddon, send resize and call backend pty.resize.
Color codes look like garbage text You are looking at ANSI in logs, not rendered screen output \u001b[31m is a control sequence; xterm.js parses and renders it.
node-pty fails to start Node version, native build, helper permissions, shell path It is not a pure JS package; on macOS, spawn-helper needs execute permission.
Backend shell remains after closing the page WebSocket / PTY cleanup is missing Frontend calls ws.close(); backend calls ptyProcess.kill().

Final Memory Hook

xterm.js is a terminal emulator, not a shell. In a real Web Terminal, the frontend only sends input to the backend, writes backend output back to the terminal, syncs size, and manages resources. Command parsing, prompts, history, completion, Ctrl+C, and vim/top-style interaction come from the shell, TTY, and PTY.

xterm.js onData -> WebSocket -> node-pty.write -> shell -> pty.onData -> WebSocket -> term.write
